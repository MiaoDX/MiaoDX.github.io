March 2026 — Dual Agent Collaboration Launch ​

This month, I built a dual Agent collaboration system from 0 to 1.
Not smooth sailing, but every step had value.

Timeline ​

DateEvent

Key Milestones ​

Agent Launch & Division ​

  • Deployed WLB (🦞) and GSD (🥷⚡) two OpenClaw instances
  • Established division: WLB for decision/balance, GSD for execution/delivery
  • Established Slack #copycat channel as collaboration space

Pitfalls & Fixes ​

  • Azure Config Incident: Mistakenly set primary → gateway loop crash → restored original config
  • WeChat Scraping Blocked: IP blocked → degraded to search + manual feeding
  • 6-Hour Offline: stale lock → three-layer protection architecture (L1 Railway restart / L2 cron watchdog / L3 cross-instance heartbeat)
  • API Key Exposure: Accidentally sent in group chat → 7-minute emergency response → security rules written into protocol

Protocol Evolution ​

  • v1.0: Basic role definition
  • v1.1: Anti-loop mechanism (four-layer protection)
  • v1.2: 👀 Reaction mechanism
  • v1.3: Thread workflow
  • v1.4: Security rules (after API Key incident)
  • v1.5: Task claim rules (after Git push conflict)

LIP Content Output ​

  • Stories 6 articles: Dual Agent launch, Gateway outage, WeChat battle, API Key incident, collaboration protocol, Lobster Civilization
  • Lessons 4 articles: Cross-instance collaboration, Error→Skill, Gateway resilience, Azure config
  • Share 5 materials: OpenClaw sharing v3, low-cost deployment, Claws Civilization, Raise Lab, Tailscale config

What I Learned ​

  1. Clear division: WLB responsible for decision/architecture, GSD responsible for execution/delivery
  2. Test first: Test new configs first, confirm working before switching
  3. Failures first: Pitfall cases more valuable than success stories
  4. Protocol grows from errors: Upgrade rules after each incident
  5. Security is a system property: Not relying on each Agent's self-discipline, but on checks and balances
